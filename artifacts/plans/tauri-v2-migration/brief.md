# Brief — Voice to Text — Tauri v2 Big-Bang Migration

> This file is read by every executor before its task card.
> It contains only cross-cutting context that applies to most tasks in this migration.

## Evidence Summary
- **Structure**: Current code cleanly separates privileged shell behavior from renderer logic. `electron/main.js` owns tray, windows, global shortcut, config/credential handoff, permissions, insertion, and LLM calls (`electron/main.js:105-362`); renderer files own STT, stop-word logic, state machine, and local UI state (`ui/stt.js:38-158`, `ui/bar-renderer.js:194-370`, `ui/renderer.js:8-232`). Keep the same split in Tauri: Rust commands/events for privileged behavior, renderer for UI/Web Audio/WebSocket. — source: direct reads, oracle
- **Frontend delivery**: The app already uses static HTML/CSS/JS pages (`ui/index.html:1-104`, `ui/setup.html:1-72`, `ui/bar.html:1-48`). Tauri v2 supports static `frontendDist` plus `app.withGlobalTauri: true`, so the migration can keep the `ui/` folder as the frontend without introducing Vite unless a task proves it necessary (`artifacts/libs/tauri-v2-vanilla-html-js_20.25_26-03-2026.md:8-28`, `:44-88`, `:183-198`). — source: direct reads, librarian
- **Error handling**: Current privileged flows return explicit result objects instead of silent failures: permissions return `{granted,status?,code?,openedSettings?,message?}` (`electron/macos-permissions.js:39-85`), insertion returns `{success,error?,code?,openedSettings?}` (`electron/main.js:313-333`), clipboard restore is guaranteed in `finally` (`electron/text-inserter.js:120-137`), and LLM responses are shape-validated before dereference (`electron/llm-service.js:119-133`). Preserve those explicit error/result shapes in Tauri commands. — source: direct reads
- **Security boundary**: Electron currently narrows renderer authority via a preload bridge and sender URL allowlist (`electron/preload.js:3-54`, `electron/main.js:73-103`). In Tauri, replace that with a narrow `window.voiceEverywhere` bridge plus capabilities/permissions instead of broad direct frontend access. — source: direct reads, librarian
- **Testing**: Current automated verification uses `node --test` (`package.json:6-11`) with `node:assert/strict` and lightweight VM/DOM stubs for renderer behavior (`tests/bar-renderer.test.js:25-217`). Continue JS-side behavior tests under `tests/`, and add `cargo test` coverage for Rust-side services introduced by the migration. TDD remains Red → Green → Refactor unless the task is cleanup-only or config-only. — source: direct reads
- **State persistence**: UI preferences and vocabulary lists live in `localStorage` (`ui/renderer.js:8-21`, `ui/bar-renderer.js:30-48`), while credentials resolve from app storage, inherited env, then startup-shell env (`electron/credentials.js:84-98`, `electron/shell-credentials.js:14-82`). Preserve localStorage for UI preferences; preserve all three key sources even though fresh-app cutover and new identity are allowed. — source: direct reads, interview
- **STT protocol**: Soniox must receive exactly one JSON init frame followed by binary audio only (`ui/stt.js:6-9`, `:116-149`). Do not change this protocol while migrating shells. — source: direct reads
- **Window behavior**: The bar window is intentionally kept shown and CSS-hidden to preserve all-workspaces behavior (`electron/main.js:185-191`, `:216-226`, `ui/styles.css:106-115`). Preserve the behavioral intent even if the implementation uses Tauri window APIs instead of Electron IPC no-ops. — source: direct reads
- **Bundle config**: Current build/test flow treats packaged macOS behavior as authoritative: `install.sh` validates bundle identity and mic entitlements before copying to `/Applications` (`install.sh:31-73`), and package tests assert entitlement wiring (`tests/package-build-config.test.js:12-47`). Preserve that packaged-app standard in the Tauri version. — source: direct reads

## Research Context
### Library Deep Dives (from librarian)
- **Tauri static frontend + vanilla JS**: Tauri v2 can serve the existing `ui/` folder directly via `build.frontendDist`, and `app.withGlobalTauri: true` exposes `window.__TAURI__.core` / `window.__TAURI__.event` for plain JS without a bundler (`artifacts/libs/tauri-v2-vanilla-html-js_20.25_26-03-2026.md:8-28`, `:44-88`, `:183-198`). Implementation implication: centralize all Tauri API usage in `ui/tauri-bridge.js`; do not force Vite unless the task proves static delivery impossible.
- **Tauri window/tray capabilities**: System tray, always-on-top, skip-taskbar, visible-on-all-workspaces, and ignore-cursor-events are supported via Tauri v2 core/features/plugins and capabilities (`artifacts/libs/tauri-v2-electron-migration_19.48_26-03-2026.md:53-68`, `:131-164`, `:171-233`, `:407-425`). Implementation implication: tasks must wire the correct capabilities instead of assuming Electron-like defaults.
- **No built-in mic/accessibility permission plugins**: Tauri does not provide a dedicated microphone or macOS accessibility permission plugin; microphone capture should stay in the renderer, while accessibility and AppleScript flows need custom Rust/native handling (`artifacts/libs/tauri-v2-electron-migration_19.48_26-03-2026.md:299-327`). Implementation implication: do not move audio capture native-side; do port permission coordination and paste execution into Rust/backend code.
- **macOS bundle metadata for TCC**: Tauri expects `src-tauri/Info.plist` for usage descriptions and `src-tauri/Entitlements.plist` configured through `tauri.conf.json` for entitlements (`artifacts/libs/tauri-v2-macos-bundle-config_20.21_26-03-2026.md:8-39`, `:42-84`, `:206-302`). Implementation implication: replace Electron’s `build/entitlements*.plist` with Tauri-native plist files instead of trying to reuse Electron bundle config.

### Architecture Rationale (from oracle)
- **Shell replacement is the hard part**: The renderer already uses browser-native APIs, while the risky behavior is concentrated in shell features such as tray lifecycle, window control, AppleScript paste, permissions, and packaging. Executors should therefore prefer minimal renderer churn and move complexity into explicit Rust commands/events only where required. — what to do differently: keep UI logic recognizable; do not rewrite working STT/rendering code just because the host shell changes.
- **Packaged app is the source of truth**: For this repo, mic permission, Apple Events automation, and install flow matter more than dev-mode behavior. Executors should treat packaged `/Applications` validation as the final standard even though the user chose a big-bang release strategy. — what to do differently: document manual packaged smoke checks in Task 06 instead of claiming parity from local dev runs.

## Design Decisions
- **Target platform**: macOS only. Rejected: cross-platform/mobile because the current app is macOS-specific and the user explicitly excluded expansion.
- **Release strategy**: literal big-bang rewrite. Rejected: staged releases, even though oracle recommended them as safer, because the user explicitly chose big-bang.
- **Identity/storage compatibility**: new app identity and fresh-app cutover are acceptable. Rejected: preserving old bundle identity and storage paths. However, API key source precedence still remains app storage → env → startup shell because the user explicitly kept all current key sources.
- **Setup behavior**: Soniox is required; xAI is optional. Rejected: keeping the current “both keys required” gate because the user explicitly chose the optional-xAI flow.
- **Frontend architecture**: keep static vanilla HTML/JS and use `withGlobalTauri` + a dedicated bridge file. Rejected: adding Vite/frameworks because it is unnecessary scope for the migration goal.
- **Audio architecture**: keep microphone capture and Soniox WebSocket streaming in the renderer. Rejected: native audio capture because Tauri has no built-in microphone plugin and the current renderer path already fits the webview model.
- **Packaging model**: use `src-tauri/Info.plist` + `src-tauri/Entitlements.plist` + Tauri bundle config. Rejected: carrying forward Electron’s `build/entitlements.mac*.plist` structure.

## Pattern Conflicts
- **Setup requirements conflict**: current setup UI requires both keys (`ui/setup.js:6-27`), while README calls xAI optional (`README.md:34-35`). Chosen pattern: xAI optional, Soniox required — because the user explicitly confirmed that change.
- **Security boundary mechanism conflict**: current shell uses Electron preload + sender URL allowlist (`electron/preload.js:3-54`, `electron/main.js:73-103`), while Tauri’s model is capabilities + command exposure. Chosen pattern: Tauri capabilities + narrow `window.voiceEverywhere` bridge.
