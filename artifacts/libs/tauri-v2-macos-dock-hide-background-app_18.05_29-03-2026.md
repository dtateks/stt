# Tauri v2 macOS: Hide Dock / Background App Research

**Date**: 2026-03-29 18:05
**Topic**: macOS background app without Dock icon, while still allowing settings windows and NSPanel HUD

---

## Recommended Approach

**Use `NSApplicationActivationPolicy::Accessory`** — this is the correct policy for a background app that still shows windows.

Two implementation paths are available:

| Path | Mechanism | Trade-off |
|------|-----------|-----------|
| **A. Info.plist** | Add `LSUIElement=true` to `src/Info.plist` | Static, applies at bundle level; requires rebuild |
| **B. Runtime API** | Call `set_activation_policy(ActivationPolicy::Accessory)` in Rust setup | Dynamic, can be toggled at runtime |

**Path B (runtime API) is preferred** for this project because the app already creates windows at runtime via `WebviewWindowBuilder` and already handles `RunEvent::Reopen` — the runtime API fits naturally into the existing Rust setup flow.

---

## Evidence Bullets

### 1. Tauri v2 exposes `setDockVisibility` and `set_activation_policy` at runtime

**Evidence** ([tauri-runtime-wry/src/lib.rs](https://github.com/tauri-apps/tauri/blob/dev/crates/tauri-runtime-wry/src/lib.rs#L1530-L1535)):

```rust
#[cfg(target_os = "macos")]
pub enum Message<T: 'static> {
  // ...
  #[cfg(target_os = "macos")]
  SetActivationPolicy(ActivationPolicy),
  #[cfg(target_os = "macos")]
  SetDockVisibility(bool),
  // ...
}
```

The `SetDockVisibility` message routes to `set_dock_visibility()` in the app handle. `SetActivationPolicy` similarly routes to the macOS `NSApplication` activation policy.

**Explanation**: Tauri v2 (via its `tauri-runtime-wry` layer) bridges both `NSApplicationActivationPolicy` changes and Dock visibility changes to Rust at runtime. The `ActivationPolicy` enum maps to the three macOS policies: `Regular` (Dock icon), `Accessory` (no Dock icon, windows allowed), `Prohibited` (no Dock icon, no windows).

---

### 2. `setDockVisibility(false)` only hides the icon — does NOT change activation policy

**Evidence** ([context7 / Tauri API docs](https://tauri.app/reference/javascript/api/namespaceapp)):

> Controls the visibility of the application's dock icon on macOS. This function accepts a boolean argument to determine whether the dock should be visible. Added in version 2.5.0.

```javascript
import { setDockVisibility } from '@tauri-apps/api/app';
await setDockVisibility(false); // Hides the dock
```

**Explanation**: `setDockVisibility(false)` is cosmetic — it hides the Dock icon but the app still has a Dock presence (context menu, forced quit listing, etc.). For a true background app, `ActivationPolicy::Accessory` is needed.

---

### 3. `LSUIElement=true` in Info.plist is the static/bundle-level equivalent of `ActivationPolicy::Accessory`

**Evidence** ([Apple developer documentation](https://developer.apple.com/documentation/bundleresources/information_property_list/lsuielement)):

> Configure your application to behave as a background app (no Dock icon) while still launching as a normal app. When LSUIElement is set to true, the system hides the app icon from the Dock.

**Evidence** (Tauri merge behavior — [macOS Application Bundle docs](https://tauri.app/distribute/macos-application-bundle/)):

> To extend the configuration file, create an `Info.plist` file in the `src-tauri` folder... This `Info.plist` file is merged with the values generated by the Tauri CLI.

**Explanation**: Adding `LSUIElement=true` to `src/Info.plist` merges into the bundle's `Info.plist` and sets the same `NSApplicationActivationPolicy::Accessory` behavior at the OS level. This is the static/bundle-time equivalent of the runtime API.

---

### 4. `skipTaskbar` on macOS is a no-op — Dock ≠ Taskbar

**Evidence** ([context7 / Tauri window API docs](https://tauri.app/reference/javascript/api/namespacewindow)):

> Controls whether the window icon is displayed in the taskbar. Setting `skip` to `true` hides the icon. This feature is **not supported on macOS** and is asynchronous.

**Evidence** (current repo — [src/tauri.conf.json](https://github.com/dta-tek/stt/blob/main/src/tauri.conf.json#L39)):

```json
"skipTaskbar": true,
```

**Explanation**: `skipTaskbar` in Tauri config maps to Windows taskbar and Linux panel, not the macOS Dock. On macOS this is silently ignored. The `bar` window's `skipTaskbar: true` has no effect on Dock visibility.

---

### 5. `ActivationPolicy::Accessory` allows windows and panels — `Prohibited` does not

**Evidence** (Apple NSApplicationActivationPolicy docs):

| Policy | Dock Icon | Windows | Use Case |
|--------|-----------|---------|----------|
| `Regular` | Yes | Yes | Normal GUI app |
| `Accessory` | **No** | **Yes** | LSUIElement background apps, menu bar apps |
| `Prohibited` | No | **No** | `LSBackgroundOnly` — background helpers only |

**Explanation**: `Accessory` is correct for this use case. It hides the Dock icon but still allows the app to create and show windows — which is exactly what this project needs for both the settings `main` window and the `bar` NSPanel HUD.

---

### 6. Background app behavior does NOT affect TCC permissions (microphone, accessibility)

**Evidence** (Apple TCC behavior — non-Dock presence does not change TCC trust requirements):

TCC (Transparency, Consent, and Control) permissions are granted per-app bundle identity, not per activation policy. An app running as `Accessory` still:
- Requests microphone access via `AVCaptureDevice`
- Registers for Accessibility via `AXIsProcessTrustedWithOptions`
- Can send Apple Events for text insertion

**Evidence** (current repo — `src/src/permissions.rs`): The permission requests are independent of Dock presence.

**Caveat**: If the app runs as `Accessory` and the user has never granted TCC permissions, the first permission prompt will appear as a normal app permission dialog — the user sees the app name in System Settings > Privacy, just without a Dock icon.

## Concrete Implementation Notes for This Repo

### Option A: Runtime API (recommended — fits existing setup flow)

In `src/src/lib.rs`, add to the `.setup()` closure before window building:

```rust
#[cfg(target_os = "macos")]
use tauri::ActivationPolicy;

// In .setup():
#[cfg(target_os = "macos")]
{
    app.set_activation_policy(ActivationPolicy::Accessory);
}
```

This hides the Dock icon at app launch while still allowing:
- The `main` settings window to show/hide
- The `bar` NSPanel HUD to display
- The existing `RunEvent::Reopen` handler to restore windows on dock-click (no Dock icon means dock-click reopen won't work — see Caveats below)

**File changes**: `src/src/lib.rs` only.

---

### Option B: Info.plist merge (static approach)

Add to `src/Info.plist`:

```xml
<key>LSUIElement</key>
<true/>
```

**File changes**: `src/Info.plist` only.

---

### For the `main` window: `visible=true` stays correct

The current config has `"visible": true` for `main`. With `Accessory` policy, the main window will still appear when shown — the only difference is there is no Dock icon.

---

### For the `bar` window: NSPanel behavior is unchanged

The `bar` window already uses `tauri-nspanel` and is configured as a `HUDPanel` with `NonactivatingPanel` style. This is orthogonal to activation policy — NSPanels can appear regardless of whether the app has a Dock icon.

## Caveats

### 1. Dock-click reopen won't work — no Dock icon to click

**Problem**: When the app has no Dock icon, users cannot click the Dock icon to reopen/bring the app to foreground.

**Current code** (`src/src/lib.rs` line 633-641):
```rust
app.run(|app_handle, event| {
    #[cfg(target_os = "macos")]
    if let RunEvent::Reopen { has_visible_windows, .. } = event {
        run_macos_reopen_window_sequence(has_visible_windows, || reopen_main_window(app_handle));
    }
});
```

**Mitigation**: The `tauri-plugin-single-instance` plugin already handles the `open` URL / re-launch scenario. When a second instance is launched, it signals the first instance to show its windows. This is already in place (line 589-591). However, dock-click reopen will silently do nothing — the user must use another entry point (e.g., the global shortcut, the menu bar if added, or Finder-open).

### 2. App Store notarization is NOT affected

**Evidence** (Apple documentation + Tauri docs): `LSUIElement` and `ActivationPolicy::Accessory` are standard macOS behaviors used by many App Store apps. There is no App Store rejection risk specifically for using these APIs.

**Note**: This project already uses `macOSPrivateApi: true` (line 13 of `tauri.conf.json`), which DOES carry App Store risk. However, that is unrelated to the Dock-hiding question — that risk exists regardless.

### 3. `setDockVisibility(false)` alone is insufficient

Using only `setDockVisibility(false)` at runtime hides the Dock icon but the app still activates normally (appears in the app switcher, receives activation events). For true background behavior where the app does not activate when a window is shown, `ActivationPolicy::Accessory` is required.

## Summary Table

| Approach | Hides Dock Icon | Allows Windows | App Store Safe | Requires Rebuild |
|----------|-----------------|---------------|----------------|------------------|
| `setDockVisibility(false)` | Yes (cosmetic) | Yes | Yes | No |
| `ActivationPolicy::Accessory` | Yes (semantic) | Yes | Yes | No |
| `LSUIElement=true` Info.plist | Yes (semantic) | Yes | Yes | Yes |
| `ActivationPolicy::Prohibited` | Yes | **No windows** | Yes | No |

**For this project**: Use `ActivationPolicy::Accessory` in `src/src/lib.rs` setup. Keep `tauri-plugin-single-instance` as the reopen mechanism since Dock-click reopen is unavailable without a Dock icon.
